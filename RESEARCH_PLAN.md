# Research Plan: Empirical Characterization of Agent-Ready MCP Tool Synthesis

## Motivation

LLMs can synthesize MCP tool implementations from API documentation. But "compiles and runs" is not the same as "an agent can successfully use this to complete real tasks." The gap between the two — and what drives it — is the core research question.

The practical goal is to support more autonomous agents by understanding what makes synthesized tools reliably usable. The research contribution is an empirical characterization of this fidelity gap: how large is it, what tool properties drive it, and how does synthesis quality vary across model tiers and API complexity?

---

## Research Question

**Given identical tasks and a fixed agent, how much does synthesized tool quality determine success — and what specific tool properties drive the variance?**

Secondary: How does the fidelity gap between synthesized and human-authored tools vary across API domains, model tiers, and task complexity?

---

## Design

### Phase 1: Tool Synthesis (unchanged from current pipeline)

- Synthesize MCP server implementations from API documentation using `synthesize_all.py`
- Models produce `generated_tools/` package + `server.py` only — **no SCENARIO_SOLUTIONS.md**
- Scenario descriptions are withheld from the synthesis prompt entirely, so the model is not anchored to specific workflows during implementation
- Multiple model tiers synthesize the same APIs to enable cross-model comparison

### Phase 2: Baseline (Human-Authored Tools)

- For a subset of datasets, produce human-authored reference implementations of the same APIs
- These serve as the upper-bound comparison for the fidelity gap measurement
- Candidate sources: existing open-source MCP servers, hand-written implementations

### Phase 3: Agent Evaluation

- A fixed agent (ReAct loop, e.g. Claude or GPT-4o with tool use) attempts tasks against each tool set
- The agent receives only the task description and the list of available tools — no other context
- Tasks are drawn from the existing scenario descriptions in each dataset's TASK.md, which were generated by an LLM with access to API docs and reflect common, realistic workflows
- The same task set is used for both synthesized and human-authored tool evaluation, making the comparison fair regardless of task quality
- Task completion is measured against ground truth: expected final API state, presence of required tool calls, or verifiable output

### Phase 4: Tool Property Characterization

- For failing agent runs, attribute the failure to specific tool properties:
  - **Schema quality**: parameter names, types, descriptions — can an agent infer correct usage?
  - **Error handling**: does the tool return structured errors or raise exceptions?
  - **Response structure**: are returned IDs and fields usable for chaining subsequent calls?
  - **Coverage**: does the tool set expose what the agent needs to complete the task?
  - **Path correctness**: are base URLs, path segments, and auth patterns correct?
- Correlate these properties with agent success rate across datasets and models

## Key Design Choices and Rationale

### Why withold scenarios from synthesis?

The previous approach gave the model scenario descriptions during synthesis, anchoring its implementation to specific workflows. This biases both the tools (over-indexing on scenario paths) and the evaluation (the model already "knows" what will be tested). Withholding scenarios decouples synthesis from evaluation, making the benchmark more discriminating and the research question cleaner.

### Why reuse existing scenario descriptions as tasks?

Designing tasks from scratch is expensive and introduces its own biases. The existing scenario descriptions are grounded in real API documentation and reflect LLM priors about common workflows for well-known APIs (GitHub, Stripe, Jira). For a **comparative** study — synthesized vs. human-authored tools, same tasks — the task quality affects both conditions equally, so the fidelity gap measurement remains valid. The limitation (tasks may underrepresent edge cases) is acknowledged and does not invalidate comparative claims.

### Why a fixed agent rather than scripted scenario code?

The previous evaluation exec'd model-generated scenario code to test tools, which conflated two distinct skills: tool implementation quality and scenario-writing quality. A model producing a correct tool but a malformatted scenario doc scored 0%. A fixed third-party agent eliminates this confound — the agent's behavior is constant across conditions, so variance in task completion is attributable to tool quality.

### Why not mocks for agent evaluation?

The existing mock layer was designed for scripted scenario evaluation: fixed call sequences, predictable paths. An agent loop makes calls in unpredictable order, may retry, explores dead ends, and chains dynamic IDs across calls. Mocks that handle this accurately are essentially re-implementing the API. Real test APIs (Stripe test mode, GitHub test repos, Jira sandbox) are more practical and produce more credible results. The mock infrastructure built in the previous phase is preserved but deprioritized for the agent evaluation loop.

### Why is this not saturated?

Current results on mock-supported datasets with SOTA models reach 85–100% when tools are correct and scenarios are well-formatted. But this measures a proxy. Agent task completion with synthesized tools on realistic multi-step tasks is a harder and less saturated problem — models that produce syntactically correct tools still fail on response structure, error handling, and cross-call chaining in ways that block agent progress.

### What makes this a research contribution rather than an engineering project?

The benchmark itself is not the contribution. The contribution is:
1. An empirical taxonomy of tool properties that predict agent success — this does not currently exist in the literature
2. A quantified fidelity gap between synthesized and human-authored tools across API domains and model tiers
3. A reproducible methodology for measuring agent-readiness of tool sets

Methods contributions (e.g. iterative refinement from agent failures, synthesis quality critics) are explicitly left for future work.

## What Carries Over from Current Work

- Synthesis pipeline (`synthesize_all.py`, `synthesize.py`, pychomsky integration)
- Dataset structure and API documentation per dataset
- MCP server infrastructure (FastMCP, server.py pattern)
- Mock infrastructure (preserved, not primary for agent eval)
- Evaluation results from the current synthesis round as a baseline data point

## What Changes

- Synthesis prompt: remove SCENARIO_SOLUTIONS.md requirement
- Evaluation layer: replace scenario script execution with agent loop
- New component: agent harness (ReAct loop, task runner, completion verifier)
- New component: tool property analyzer (schema quality metrics, coverage checker)
- New component: human-authored reference implementations for a subset of datasets
